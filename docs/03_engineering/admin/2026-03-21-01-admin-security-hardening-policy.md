# admin セキュリティ強化方針メモ

## 目的
- `/admin` の保護方針を、現時点の運用規模に合わせて整理する。
- 過剰実装を避けつつ、必要十分な対策を決める。

## 現在の方針
- `/admin` は通常ログインに加えて `role=admin` のみアクセス可能にする。
- 非管理者アクセス時は `root_path` にリダイレクトする。
- 非管理者アクセス時は flash を出さない。
- セキュリティのため、権限不足の理由など詳しい情報は返さない。

## 現時点で採用する対策
### 1. アプリ側の認証・認可
- `Authentication` concern を使って未ログインを遮断する。
- `User` の `role` を `member` / `admin` の string enum で管理する。
- `Admin::ApplicationController` で `Current.session.user.admin?` を確認する。

### メリット
- 実装がシンプル。
- 個人開発でも運用しやすい。
- `/admin` の保護として必要最低限を満たせる。

### デメリット
- `/admin` の URL 自体は見える。
- 本番 console に入れる運用者の誤操作までは完全には防げない。

## 今は採用しない対策
### 1. admin 付与メールアドレスの allow-list
#### 内容
- `role=admin` にできるメールアドレスを ENV や validation で限定する。

#### メリット
- 誤って別ユーザーを admin にする事故を減らせる。

#### デメリット
- 不正アクセス者が本番 console に入れた時点で、allow-list 自体も書き換えられるため、強い防御にはなりにくい。
- 運用コストの割に効果が限定的。

#### 判断
- 現時点では見送る。
- 必要なら将来、誤操作防止目的で追加検討する。

### 2. IP 制限
#### 内容
- `/admin` を特定 IP からのみアクセス可能にする。

#### メリット
- 認証・認可に加えて、到達元ネットワークでも絞れる。
- 防御層としては有効。

#### デメリット
- 個人開発では運用負荷が高い。
- 回線変更や設定ミスで自分が入れなくなるリスクがある。
- 現段階ではオーバースペックになりやすい。

#### 判断
- 現時点では見送る。
- 運用体制や公開範囲が変わったら再検討する。

## 結論
- 現在は「通常ログイン + `role=admin` の認可」で運用する。
- 詳しい拒否理由は返さず、シンプルな遮断に留める。
- allow-list や IP 制限は、個人開発の現段階では導入しない。
