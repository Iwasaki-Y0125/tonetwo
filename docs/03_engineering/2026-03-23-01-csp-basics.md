# CSP概要メモ

## 目的
- `Content-Security-Policy` の役割を、導入前に短く把握できるようにする。
- 実装判断の前提になる一次ソースをまとめる。

## CSP とは
- CSP は、ブラウザに対して「このページが読み込んでよいリソース元」を指示する仕組み。
- 主な目的は XSS の被害抑制で、想定外の script 実行や外部リソース読み込みを制限する。
- アプリのバグを完全に防ぐものではなく、HTML エスケープや安全なテンプレート実装と併用する前提の追加防御。

## 何を制御するか
| directive | 何を制御するか | たとえば何に効くか |
| --- | --- | --- |
| `default-src` | 個別指定していない読み込み先の共通ルール | 画像やフォントなどで個別ルール未設定時の基準 |
| `base-uri` | `<base>` タグでどの URL を基準にしてよいか | 相対 URL の基準を外部サイトに変えさせない |
| `img-src` | 画像をどこから読み込んでよいか | 自サイト画像だけ許可する、`data:` 画像を許可する |
| `object-src` | `object` / `embed` / `applet` を使ってよいか | 古い埋め込み系機能を禁止する |
| `script-src` | JavaScript をどこから読み込み、実行してよいか | 自サイトの JS だけ許可する、inline script を禁止する |
| `style-src` | CSS をどこから読み込んでよいか | 自サイトの CSS だけ許可する、inline style を禁止する |
| `connect-src` | API 通信先をどこまで許可するか | `fetch`、XHR、WebSocket の接続先制御 |
| `frame-src` | iframe などの埋め込み先をどこまで許可するか | 外部サービスの埋め込み許可 |

## どう許可を書くか
| 書き方 | 意味 | 使いどころ |
| --- | --- | --- |
| `'self'` | 同一オリジンのみ許可 | 基本は自サイト配信だけにしたいとき |
| `'none'` | 何も許可しない | 明示したもの以外を禁止したいとき |
| `'unsafe-inline'` | inline script / style を許可 | 既存実装の都合で一時的に残すとき |

補足:
- `default-src 'self'` は「個別指定していない読み込み先は、基本的に同一オリジンだけ許可」という意味。
- `default-src 'none'` を使うと、必要な directive を個別に明示していく形になる。
- `'unsafe-inline'` は XSS 耐性を弱めやすいため、可能なら避ける。
- nonce は「この inline script / style は正規のもの」とブラウザへ伝えるための印。

## 導入モード
| モード | 意味 | 使いどころ |
| --- | --- | --- |
| `Content-Security-Policy` | 違反を実際に遮断する | 許可元の確認が終わっている段階 |
| `Content-Security-Policy-Report-Only` | 違反を検知するが遮断しない | 初回導入や影響調査の段階 |

補足:
- `Report-Only` は違反の有無を観測したいときの安全な導入モード。

## `default-src` がカバーするもの / しないもの
`default-src` は何でも受け持つわけではなく、主に「リソース読み込み系」の directive の fallback として働く。

### `default-src` が fallback する主な directive
| directive | 何を制御するか | 実務上の見方 |
| --- | --- | --- |
| `connect-src` | API 通信や WebSocket の接続先 | `fetch` や XHR の送信先を絞りたいときに使う |
| `font-src` | フォントの読み込み先 | Web フォントを使うときに必要になる |
| `frame-src` | iframe などで埋め込む先 | YouTube や外部ウィジェットを埋め込むときに見る |
| `img-src` | 画像の読み込み先 | 通常画像や `data:` 画像をどこまで許可するか決める |
| `manifest-src` | Web App Manifest の読み込み先 | PWA を使うときに関係する |
| `media-src` | 音声や動画の読み込み先 | audio / video を配信するときに使う |
| `object-src` | `object` / `embed` / `applet` の読み込み先 | 使わないなら `none` を明示しやすい |
| `script-src` | JavaScript の読み込み元と実行条件 | CSP の中心で、JS の許可元を決める |
| `script-src-elem` | `<script>` 要素で読み込む JavaScript | `script-src` を要素読み込みだけ細かく分けたいときに使う |
| `script-src-attr` | HTML 属性内の JavaScript | `onclick` など属性ベースの JS を制御するときに使う |
| `style-src` | CSS の読み込み元と適用条件 | CSS 全体の許可元を決める |
| `style-src-elem` | `<style>` 要素や `<link rel=\"stylesheet\">` の CSS | CSS 読み込みだけ細かく分けたいときに使う |
| `style-src-attr` | `style=\"...\"` のような inline style 属性 | style 属性だけを個別に制御したいときに使う |
| `worker-src` | Web Worker や Service Worker の読み込み先 | Worker 系を使うときに必要になる |

補足:
- これらを個別に書かなければ、`default-src` の値が使われる。
- たとえば `font-src` を書かなければ、フォント読み込みにも `default-src` が効く。

補足:
- `style-src-elem` は `<style>` 要素や `<link rel="stylesheet">` をブラウザが評価するときの見方。
- `style-src` を個別に書かなければ `default-src` が fallback する。
- ただし、nonce 付き `<style>` を許可したい場合は、nonce を載せる先として `style-src` を明示する必要がある。

### `default-src` が fallback しない代表例
| directive | 何を制御するか | 実務上の見方 |
| --- | --- | --- |
| `base-uri` | `<base>` タグで相対 URL の基準にしてよい URL | 使わないなら `none` で閉じやすい |
| `form-action` | フォームの送信先 | 外部サイトへの form 送信を防ぎたいときに明示する |
| `frame-ancestors` | 自分のページをどの親ページが iframe 埋め込みしてよいか | クリックジャッキング対策で `none` を入れやすい |
| `report-uri` | CSP 違反レポートの送信先 | 違反レポートの受け口を作るときに使う |
| `sandbox` | ページや埋め込みコンテンツに強い制限をかける | 通常ページ全体には強すぎることが多い |

補足:
- これらはリソース読み込み先ではなく、文書の振る舞い、埋め込み、送信先などを制御する directive。
- たとえば `base-uri` は `default-src` では制限されないため、制限したいなら個別に明示が必要。

## 最小構成の見方
たとえば次のような方針は、「まず必要最小限だけを許可する」という考え方に沿う。

```http
Content-Security-Policy: default-src 'self'; object-src 'none'; base-uri 'self';
```

この例の意味:
- 基本の読み込み元は同一オリジンに限定する。
- `object` 系リソースは許可しない。
- `<base>` タグの書き換え元を同一オリジンに限定する。

ただし、実際には画像、フォント、外部 API、importmap、分析タグなどの有無で `img-src`、`font-src`、`connect-src`、`script-src` を追加調整する。
また、nonce 付き `<style>` を使うライブラリがある場合は `style-src` も個別に明示する。

## 導入時の実務ポイント
- 先にインライン JavaScript を減らすと、`unsafe-inline` を避けやすい。
- アプリ側の inline style を除去していても、ライブラリが runtime で `<style>` を挿入することがある。
- いきなり広く許可するより、まず最小構成で違反箇所を把握する方が調整しやすい。
- 初回は `Report-Only` で違反を確認し、その後 enforce に切り替える流れが取りやすい。
